Project Scoping

It's really important that everyone properly understands a project before committing to it. We need to make sure we have an accurate grasp of the risks and parameters of a project. Are we the best people to do it? Has it already been done elsewhere? Can we do a good job given the budget or time constraints?

This phase is designed to answer those questions, and make sure everyone sets off in the right direction.

Project scoping involves the following activities

  • New Project Request
  • Project Scoping
  • The Estimate
  • Go / No Go decision

New Project Request

New projects should start with the new projects request form

This is a quick structured way for us to collect all the key initial details about your project such as timeframes, outline budgets and stakeholders

Completing the form will trigger the Head of Informatics to assign a product manager to the project, and arrange a meeting with you to learn a bit more about the project, and ultimately whether we can help.

Project Scopting Meeting

The project scoping meeting is generally an hour long meeting with you to understand your project in more detail.

The goal of this meeting is to:

  • Understand how developed the concept is
  • Talk about timescales & budgets vs expectations of clients
  • Identify key stakeholders
  • Identify risks
  • Identify dependencies and limitations (eg. needs to be written in Java and delivered for the COP in September)
  • Is there anything else out there that does the same/similar?
  • Explain our processes and why we work this way
  • Set the relationship expectations (e.g will stakeholders be available for feedback)

From this information we'll be able to develop an outline time and cost estimate.

The Estimate

Following the project scoping meeting the Informatics team will have enough information to provide an outline estimate for the work.

The "Go / No Go decision"

This is the last stage in the scoping phase.

We really want to work on as many great projects as possible. However, it's really important that we work on projects that are going to deliver value, and under conditions in which we have a sporting chance of doing a great job.

Therefor, there are some circumstances where we might advise working with another team, or not doing the project at all.

For example:

  • It might require the use of a specialist technology we're not familiar with
  • The problem might already have been solved by someone else
  • Other teams might be better placed to do the work depending on the conditions of the project
  • We might feel the time or budget constraints mean it's unlikly we'll be able to complete the project successfully

If we don't think it's a good idea to work on a project, we'll help you find someone who can, or work with you to find alternative solutions
